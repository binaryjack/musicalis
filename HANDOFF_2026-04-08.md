## Critical Bug Fix: Chord Vertical Drag

### Problem
**Issue**: Cannot drag a note vertically to change pitch when the beat contains a chord (multiple notes).

**Example**: In a C-E-G chord, dragging C4 to D4 was blocked.

### Root Cause
The behavior tree's drag commit sequence had a guard decorator that prevented ANY drop operation when notes existed at the target beat:

```typescript
{
  id: 'd-drag-guard', type: 'decorator', label: 'No Note At Target', decoratorType: 'invert',
  children: [{ id: 'c-drag-occupied', type: 'condition', label: 'Position Has Note', conditionKey: 'position.hasNote' }],
}
```

This guard blocked the drag **before** `handleMoveNote` could run its proper validation (`validateMeasureMatrix`), which already correctly handles chords.

### Solution
**File**: `src/features/behavior-tree/store/bt-default-tree.ts` (lines 73-86)

**Changed**: Removed the decorator guard from the `seq-drag-commit` sequence.

**Before**:
```typescript
{
  id: 'seq-drag-commit', type: 'sequence', label: 'Note Drag (commit)',
  children: [
    { id: 'c-drag-active', type: 'condition', label: 'Is Dragging',     conditionKey: 'note.isDragging' },
    { id: 'c-drag-up',     type: 'condition', label: 'Mouse Released',  conditionKey: 'mouse.isUp' },
    {
      id: 'd-drag-guard', type: 'decorator', label: 'No Note At Target', decoratorType: 'invert',
      children: [{ id: 'c-drag-occupied', type: 'condition', label: 'Position Has Note', conditionKey: 'position.hasNote' }],
    },
    { id: 'a-drag-rest-del', type: 'action', label: 'Delete Rest',  actionKey: 'rest.deleteAtPosition' },
    { id: 'a-drag-commit',   type: 'action', label: 'Commit Drag',  actionKey: 'note.commitDrag' },
    { id: 'a-drag-sound',    type: 'action', label: 'Play Sound',   actionKey: 'note.playSound' },
  ],
}
```

**After**:
```typescript
{
  id: 'seq-drag-commit', type: 'sequence', label: 'Note Drag (commit)',
  children: [
    { id: 'c-drag-active', type: 'condition', label: 'Is Dragging',     conditionKey: 'note.isDragging' },
    { id: 'c-drag-up',     type: 'condition', label: 'Mouse Released',  conditionKey: 'mouse.isUp' },
    { id: 'a-drag-rest-del', type: 'action', label: 'Delete Rest',  actionKey: 'rest.deleteAtPosition' },
    { id: 'a-drag-commit',   type: 'action', label: 'Commit Drag',  actionKey: 'note.commitDrag' },
    { id: 'a-drag-sound',    type: 'action', label: 'Play Sound',   actionKey: 'note.playSound' },
  ],
}
```

### Impact
- ✅ Users can now drag notes vertically within chords to change pitch
- ✅ Validation still works correctly via `validateMeasureMatrix`
- ✅ Prevents duplicate pitches in chords
- ✅ Allows adding new notes to existing chords

## Developer Notes

### Why the Decorator Was Wrong
The behavior tree decorator `'No Note At Target'` was designed for **placing NEW notes**, where blocking occupied positions makes sense. But it inadvertently blocked **drag operations** where we specifically WANT to allow:
1. Changing pitch within a chord (C→D in C-E-G chord)
2. Adding to chords (moving from beat 1 to beat 0 that has E-G)
3. Extracting from chords (moving C from C-E-G to empty beat)

The proper validation (`validateMeasureMatrix`) already handles all these cases correctly by:
- Allowing notes at the same start position (chords)
- Preventing mid-note interruptions
- Checking duration overlaps
- Enforcing measure boundaries

### Why Redux Migration Is Critical
Current state management is chaotic:
- Staff data: `useState` hook in `use-staff-state.ts`
- UI state: Local component state
- Settings: Redux
- Behavior Tree: Redux
- **Result**: No single source of truth, impossible to implement undo/redo

After migration:
- **All state in Redux**
- Actions logged for debugging
- Time-travel debugging via DevTools
- Foundation for undo/redo
- Easier testing (pure reducers)
